# cppseed 要件定義書

- 文書バージョン: 0.1
- 対象リリース: cppseed v0.1.0
- ステータス: Draft
- 作成日: 2026-07-13

## 1. 概要

`cppseed` は、CMakeを利用する小規模なC++プロジェクトの雛形を生成するCLIツールである。

利用者は1つのコマンドを実行するだけで、ビルド、実行、テストが可能なC++プロジェクトを作成できる。ツール本体はC++で実装し、外部ランタイムを要求しない単一の実行ファイルとして配布する。

### 1.1 解決する課題

C++プロジェクトを新規作成するとき、利用者は毎回以下の作業を行う必要がある。

- ディレクトリ構成の作成
- `CMakeLists.txt` の作成
- 実行ファイルとライブラリのターゲット定義
- テストの設定
- `.gitignore` と `.clang-format` の用意
- 最初のビルドが成功するまでの設定調整

`cppseed` はこれらの反復作業を自動化する。

### 1.2 プロダクトの価値

- 生成直後からビルドとテストを実行できる
- ツール自体の導入にPython、Rust、Node.jsなどのランタイムを必要としない
- CMakeの基本的かつ理解しやすい構成を生成する
- 利用者の既存ファイルを安全に保護する

## 2. 目標と対象外

### 2.1 v0.1.0の目標

1. 1コマンドでC++プロジェクトを生成できること
2. 生成物がmacOS、Linux、Windowsでビルドできること
3. 生成直後にCTestでテストを実行できること
4. 既存ファイルを意図せず変更または削除しないこと
5. GitHub Releasesから取得した単一の実行ファイルで動作すること
6. CIでツール本体と生成物の両方を検証できること

### 2.2 v0.1.0の対象外

以下はv0.1.0には含めない。

- Eigen、GoogleTestなどの外部依存関係の追加
- Conan、vcpkgなどのパッケージマネージャー連携
- ユーザー定義テンプレート
- 複数のプリセット
- GUIまたは対話形式の入力
- 既存ディレクトリへのファイル追加
- 既存ファイルの強制上書き
- `git init` の自動実行
- GitHubリポジトリの自動作成
- CMake以外のビルドシステム
- プロジェクト生成後の更新・移行管理

## 3. 想定利用者

### 3.1 主な利用者

- C++を学習している開発者
- 小規模なC++ツールや実験コードを頻繁に作成する開発者
- CMakeプロジェクトの最低限の雛形をすぐに用意したい開発者

### 3.2 前提知識・環境

利用者は以下をローカル環境に用意しているものとする。

- C++17以降に対応したC++コンパイラ
- CMake 3.20以降
- 生成されたプロジェクトをビルドするためのビルドツール

`cppseed` 自体を実行するためにC++コンパイラやCMakeは不要とする。ただし、生成されたC++プロジェクトのビルドには必要である。

## 4. ユースケース

### UC-01: 標準構成のプロジェクトを作成する

```console
$ cppseed new fem-solver
Created C++ project `fem-solver` in ./fem-solver

Next steps:
  cd fem-solver
  cmake -S . -B build
  cmake --build build
  ctest --test-dir build
```

### UC-02: C++規格を指定する

```console
$ cppseed new fem-solver --std 23
```

生成される `CMakeLists.txt` はC++23を要求する。

### UC-03: ヘルプを表示する

```console
$ cppseed --help
$ cppseed new --help
```

### UC-04: バージョンを表示する

```console
$ cppseed --version
cppseed 0.1.0
```

## 5. CLI仕様

### 5.1 コマンド形式

```text
cppseed new <PROJECT_NAME> [--std <17|20|23>]
cppseed --help
cppseed --version
```

### 5.2 引数とオプション

| 名前 | 必須 | 初期値 | 説明 |
|---|---:|---:|---|
| `PROJECT_NAME` | はい | なし | 作成するプロジェクト名 |
| `--std` | いいえ | `20` | 使用するC++規格。`17`、`20`、`23`のいずれか |
| `-h`, `--help` | いいえ | なし | ヘルプを表示する |
| `-V`, `--version` | いいえ | なし | cppseedのバージョンを表示する |

### 5.3 終了コード

| コード | 意味 |
|---:|---|
| `0` | 正常終了 |
| `1` | 入力、ファイル操作、その他の実行時エラー |
| `2` | コマンドまたはオプションの使用方法が不正 |

## 6. 機能要件

### FR-001: プロジェクト生成

ツールは `cppseed new <PROJECT_NAME>` により、現在のディレクトリ直下へ `<PROJECT_NAME>` ディレクトリを作成しなければならない。

### FR-002: 生成ファイル

標準構成では以下を生成しなければならない。

```text
<PROJECT_NAME>/
├── CMakeLists.txt
├── README.md
├── .clang-format
├── .gitignore
├── include/
│   └── <INCLUDE_NAME>/
│       └── hello.hpp
├── src/
│   ├── hello.cpp
│   └── main.cpp
└── tests/
    ├── CMakeLists.txt
    └── hello_test.cpp
```

### FR-003: プロジェクト名の検証

`PROJECT_NAME` は以下の条件をすべて満たさなければならない。

- 1文字以上64文字以下
- 先頭はASCII英字
- 2文字目以降はASCII英数字、ハイフン、アンダースコアのみ
- `.`、`..`、パス区切り文字、空白を含まない

概念上の検証パターンは以下とする。

```regex
^[A-Za-z][A-Za-z0-9_-]{0,63}$
```

### FR-004: 名前の正規化

入力されたプロジェクト名は用途ごとに以下のように変換する。

| 用途 | `fem-solver` を入力した場合 |
|---|---|
| 出力ディレクトリ | `fem-solver` |
| CMakeプロジェクト名 | `fem-solver` |
| 実行ファイル名 | `fem-solver` |
| C++ライブラリターゲット名 | `fem_solver_lib` |
| includeディレクトリ名 | `fem_solver` |
| C++名前空間 | `fem_solver` |

C++識別子として使う名前では、ASCII英字を小文字化し、ハイフンまたはアンダースコアが連続する区間を1個のアンダースコアへ変換する。これにより、C++で予約される二重アンダースコアを含む識別子を生成しない。

### FR-005: C++規格の指定

ツールは `--std 17`、`--std 20`、`--std 23` を受け付けなければならない。未指定時はC++20を使用する。

その他の値を指定した場合は生成処理を行わず、エラーを返さなければならない。

### FR-006: ビルド可能性

生成されたプロジェクトは、サポート対象環境において以下のコマンドで正常にビルドできなければならない。

```console
cmake -S . -B build
cmake --build build
```

### FR-007: 実行可能性

生成された実行ファイルは正常終了し、標準出力へ次を表示しなければならない。

```text
Hello from <PROJECT_NAME>!
```

### FR-008: テスト可能性

生成されたプロジェクトは外部テストフレームワークを使用せず、CTestでテストを実行できなければならない。

```console
ctest --test-dir build --output-on-failure
```

最低1件のテストを含み、正常な生成物では全件成功しなければならない。

### FR-009: 既存ディレクトリの保護

生成先と同名のファイルまたはディレクトリが存在する場合、ツールは何も上書きせずエラー終了しなければならない。

v0.1.0では上書きを許可するオプションを提供しない。

### FR-010: 失敗時のクリーンアップ

生成処理の途中で失敗した場合、ツールはその実行中に自ら作成したファイルとディレクトリを可能な限り削除しなければならない。実行前から存在したファイルまたはディレクトリを削除してはならない。

### FR-011: 成功メッセージ

生成成功時は、作成先とビルド・テストの次の操作を標準出力へ表示しなければならない。

### FR-012: エラーメッセージ

エラー時は以下を標準エラー出力へ表示しなければならない。

- `error:` で始まる原因
- 利用者が修正可能な場合は、修正方法または許可される値

例:

```text
error: directory `fem-solver` already exists
```

```text
error: invalid project name `fem solver`
hint: use 1-64 characters: ASCII letters, numbers, hyphens, or underscores
```

### FR-013: テンプレートの内包

生成に必要なテンプレートは実行ファイルへ埋め込み、通常利用時に追加のテンプレートファイルを要求してはならない。

## 7. 生成されるCMakeプロジェクトの要件

### 7.1 CMake最低バージョン

```cmake
cmake_minimum_required(VERSION 3.20)
```

### 7.2 ターゲット構成

生成物は最低限、以下のターゲットを持つ。

- `<normalized_name>_lib`: `src/hello.cpp` を含むライブラリ
- `<PROJECT_NAME>`: `src/main.cpp` を含む実行ファイル
- `<normalized_name>_test`: `tests/hello_test.cpp` を含むテスト実行ファイル

### 7.3 C++規格

C++規格はターゲット単位の `target_compile_features` で要求する。コンパイラ独自拡張は無効にする。

### 7.4 includeパス

ライブラリの公開ヘッダーは `include/<INCLUDE_NAME>/` に置き、ビルドツリーとインストール後の利用を考慮したincludeパスを設定する。

ただし、v0.1.0ではインストールルールおよびCMake Package Configの生成は必須としない。

### 7.5 警告オプション

生成物は特定コンパイラ専用の警告オプションを初期状態では強制しない。将来、コンパイラ判定を伴うオプションとして追加できる設計とする。

## 8. 非機能要件

### NFR-001: 実装言語

ツール本体はC++20で実装する。

### NFR-002: 外部依存

v0.1.0のツール本体は、C++標準ライブラリ以外の実行時およびサードパーティライブラリ依存を持たない。

ビルドおよびテスト用途のツールへの依存は許容する。

### NFR-003: 対応OS

ソースコードとCIは以下をサポート対象とする。

- macOS: Apple Silicon、Intel
- Linux: x86-64
- Windows: x86-64

初回リリースで提供するビルド済みバイナリの範囲は、リリース設計時に確定する。

### NFR-004: 対応コンパイラ

少なくとも以下のコンパイラ系列でCI検証可能なコードとする。

- Apple ClangまたはClang
- GCC
- MSVC

厳密な最低コンパイラバージョンは実装開始時にCI環境と合わせて確定し、READMEへ記載する。

### NFR-005: 性能

ローカルファイルシステム上で標準構成を生成する処理は、一般的な開発用PCで1秒以内を目標とする。ただし、ネットワークファイルシステムやセキュリティソフトなど外部要因による遅延は除く。

### NFR-006: 安全性

- 既存ファイルを上書きしない
- シェルコマンドを内部実行しない
- 入力文字列をパスとして無検証に連結しない
- 生成先を指定されたプロジェクトディレクトリ配下に限定する

### NFR-007: 可搬性

OS固有APIは原則として使用せず、ファイル操作には `std::filesystem` を使用する。

### NFR-008: 配布形態

リリース成果物は、対象OS・CPUごとの単一実行ファイルまたは単一実行ファイルを含む圧縮アーカイブとする。

利用者が `cppseed` を実行するために、C++コンパイラ、CMake、追加ランタイムを個別にインストールすることを前提としない。

### NFR-009: ライセンス

プロジェクト本体はMIT Licenseで公開する。生成物に適用されるライセンスについては、生成されるREADMEに「生成コードの利用をcppseedのライセンスで制限しない」旨を明記するか、テンプレート部分の扱いをLICENSEで明確化する。

### NFR-010: 保守性

- CLI解析、入力検証、名前変換、ファイル生成を分離する
- ファイル生成ロジックを単体テスト可能にする
- テンプレートの置換対象を明示的に管理する
- ユーザー入力をテンプレートへ無制限に展開しない

## 9. テスト要件

### 9.1 ツール本体のテスト

最低限、以下を自動テストする。

1. 有効なプロジェクト名を受理する
2. 無効なプロジェクト名を拒否する
3. ハイフンを含む名前をC++識別子へ正規化する
4. `--std` の有効値を受理する
5. `--std` の無効値を拒否する
6. 必要なディレクトリとファイルを生成する
7. テンプレートのプレースホルダーをすべて置換する
8. 既存の生成先を変更しない
9. 失敗時にcppseedが作成した不完全な生成物を除去する
10. `--help` と `--version` が終了コード0で終了する
11. 不正なCLI入力が終了コード2で終了する

### 9.2 End-to-Endテスト

CI上で以下を行う。

1. `cppseed` をビルドする
2. 一時ディレクトリにサンプルプロジェクトを生成する
3. 生成物をCMakeで構成する
4. 生成物をビルドする
5. 生成された実行ファイルを実行する
6. CTestを実行する

このテストはLinux、macOS、Windowsで実施する。

## 10. 受け入れ基準

v0.1.0は以下をすべて満たしたとき完成とする。

- [ ] `cppseed new hello` が仕様どおりのファイルを生成する
- [ ] `--std 17`、`--std 20`、`--std 23` が正しく反映される
- [ ] 生成された3種類のC++規格のプロジェクトがCIでビルドできる
- [ ] 生成されたテストがCTestで成功する
- [ ] macOS、Linux、Windowsでツール本体のテストが成功する
- [ ] 既存の同名ファイルまたはディレクトリを変更しない
- [ ] 不正入力で分かりやすいエラーと非0の終了コードを返す
- [ ] `README.md` の手順だけでインストール、生成、ビルド、テストができる
- [ ] MIT Licenseが含まれる
- [ ] GitHub ActionsのCIが成功する
- [ ] 少なくとも開発者自身の環境向けビルド済みバイナリをGitHub Releasesで公開できる

## 11. 将来候補

優先順位はv0.1.0公開後の利用経験とIssueに基づいて決める。

- `--gtest`: GoogleTest対応
- `--eigen`: Eigen対応
- `--type app|lib`: 生成するプロジェクト種別の選択
- `--preset numerical`: 数値計算向け構成
- `--preset competitive`: 競技プログラミング向け構成
- `--license`: 生成プロジェクトのライセンス選択
- `--git`: `git init` と初期コミット支援
- `CMakePresets.json` の生成
- sanitizers、clang-tidy、警告設定
- インストールルールとCMake Package Config
- ユーザー定義テンプレート
- Homebrew、Scoop、その他パッケージマネージャー対応

## 12. 実装前に確定する事項

以下は要件の中核を変えないため、実装着手時またはリリース設計時に確定する。

1. `cppseed` という名称がGitHub、主要パッケージレジストリ、既存商標と衝突しないか
2. v0.1.0で公開するビルド済みバイナリのOS・CPU一覧
3. サポートするコンパイラの具体的な最低バージョン
4. Linuxバイナリの標準C++ライブラリおよびglibc互換性方針
5. WindowsバイナリのMSVCランタイム配布方針
6. 生成されるテンプレートコードのライセンス上の扱い
